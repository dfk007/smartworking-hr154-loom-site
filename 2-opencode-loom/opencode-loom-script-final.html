Mongo Open

The updated JD moved MongoDB Atlas from a supporting requirement to the main gap you need to close. Learn the five Mongo topics below in this order and be ready to explain them at owner level, not tool-user level.

Original JD

MongoDB Atlas was present, but as one item inside a broader GCP, Cloud Run, Pub/Sub, Terraform role.

Updated JD

Mandatory deep MongoDB Atlas experience is now central: cluster scaling, replication/failover, backup/recovery, performance/indexing, and security/access.

Main takeaway: Pub/Sub and Cloud Run are no longer the differentiator. MongoDB Atlas ownership is.
Your angle: be honest that your production DB depth is stronger in PostgreSQL, then show Atlas IaC knowledge, distributed DB fundamentals, and hands-on practice proof.
Exact Learning Order
1
Atlas cluster setup and scaling
Most important start
Learn
  • Cluster tiers: M0 for practice, M30+ for real production conversations.

terraform {
  required_providers {
    mongodbatlas = {
      source  = "mongodb/mongodbatlas"
      version = ">= 1.0.0"
    }
  }
}

provider "mongodbatlas" {}

resource "mongodbatlas_advanced_cluster" "lab" {
  project_id   = var.project_id
  name         = "hr154-lab"
  cluster_type = "REPLICASET"

  replication_specs {
    num_shards = 1

    region_configs {
      provider_name = "AWS"
      region_name   = "EU_WEST_1"
      priority      = 7

      electable_specs {
        instance_size = "M10"
        node_count    = 3
      }

      analytics_specs {
        instance_size = "M10"
        node_count    = 0
      }

      auto_scaling {
        disk_gb_enabled = true
        compute {
          enabled            = true
          scale_down_enabled = true
          min_instance_size  = "M10"
          max_instance_size  = "M30"
        }
      }
    }
  }
}

resource "mongodbatlas_project_ip_access_list" "home" {
  project_id = var.project_id
  cidr_block = "YOUR_PUBLIC_IP/32"
  comment    = "ubuntu practice machine"
}

resource "mongodbatlas_database_user" "app_rw" {
  project_id         = var.project_id
  username           = "app_rw"
  password           = var.db_password
  auth_database_name = "admin"

  roles {
    role_name     = "readWrite"

Load sample data and tune queries
Performance + indexing
Commands
mongosh "$MONGODB_URI" --eval '
use appdb
db.events.drop()
const docs = []
for (let i = 0; i < 50000; i++) {
  docs.push({
    tenantId: i % 200,
    status: i % 3 === 0 ? "OPEN" : "CLOSED",
    amount: i % 1000,
    createdAt: new Date(Date.now() - i * 60000),
    expiresAt: new Date(Date.now() + 7 * 24 * 60 * 60000),
    email: `user${i}@example.com`
  })
}
db.events.insertMany(docs)
db.events.find({ tenantId: 42, status: "OPEN" }).sort({ createdAt: -1 }).limit(20).explain("executionStats")
db.events.createIndex({ tenantId: 1, status: 1, createdAt: -1 })
db.events.createIndex({ expiresAt: 1 }, { expireAfterSeconds: 0 })
db.events.find({ tenantId: 42, status: "OPEN" }).sort({ createdAt: -1 }).limit(20).explain("executionStats")
'
What to compare
  • First explain plan should show COLLSCAN or poor document scan behavior.
  • After the compound index, the winning plan should move toward IXSCAN.
  • Check Atlas Performance Advisor and match its index suggestion against what you created manually.
Owner-level explanation
  • Unindexed queries might feel acceptable with thousands of documents and then collapse under millions because scan cost grows with collection size.
  • Compound index order should follow filter selectivity and sort pattern, not guesswork.
  • TTL indexes are operationally useful for expiring sessions, logs, and temporary records without batch cleanup jobs.

Practice backup, restore, and DR
Snapshots + PITR
Atlas steps
  • Enable Cloud Backup on the cluster if your tier supports it.
  • Trigger one manual snapshot.
  • Walk through the restore screen and note that Atlas restores into a new cluster rather than overwriting the current one in place.
  • Record an example target RPO and RTO for interview answers.
Commands
mkdir -p ~/hr154-mongodb-atlas-lab/backups
cd ~/hr154-mongodb-atlas-lab/backups

mongodump --uri "$MONGODB_URI" --db appdb --archive=appdb.archive --gzip

mongorestore \
  --uri "$MONGODB_URI" \
  --nsFrom "appdb.*" \
  --nsTo "appdb_restore.*" \
  --archive=appdb.archive \
  --gzip
What to explain out loud
  • Snapshots and PITR solve different recovery windows; PITR gives finer recovery targets.
  • Backup is not enough; restore testing proves the backup is operationally useful.
  • Production DR should include restore validation, DNS or app cutover steps, and an application smoke test.

use appdb
db.events.drop()
const docs = []
for (let i = 0; i < 50000; i++) {
  docs.push({
    tenantId: i % 200,
    status: i % 3 === 0 ? "OPEN" : "CLOSED",
    amount: i % 1000,
    createdAt: new Date(Date.now() - i * 60000),
    expiresAt: new Date(Date.now() + 7 * 24 * 60 * 60000),
    email: `user${i}@example.com`
  })
}
db.events.insertMany(docs)
db.events.find({ tenantId: 42, status: "OPEN" }).sort({ createdAt: -1 }).limit(20).explain("executionStats")
db.events.createIndex({ tenantId: 1, status: 1, createdAt: -1 })
db.events.createIndex({ expiresAt: 1 }, { expireAfterSeconds: 0 })

Updated JD · Full Learning Scope

All Concepts

This tab is the full map of what to learn from updated-jd-with-mongo.md. MongoDB Atlas is still the highest-priority gap, but the JD also expects strong Terraform, cloud ownership, containers, event-driven systems, CI/CD, observability, incident response, and async engineering maturity.

Top Priority

MongoDB Atlas plus the ability to speak at architect-owner level.

Second Priority

Terraform + cloud systems design, then containers, event-driven systems, CI/CD, and observability.

Important: the updated JD is not asking for only tool knowledge. It wants ownership, architecture judgment, performance thinking, and operational maturity.
Use this tab like a checklist: if you cannot explain a card below clearly out loud, that concept is still a gap.
Priority Order
  • 1. MongoDB Atlas: cluster scaling, replication, failover, backups, indexing, security.
  • 2. Terraform + cloud ownership: modularisation, remote state, environment separation, design tradeoffs.
  • 3. Containers + serverless: Docker, Kubernetes, Cloud Run or equivalents, when to choose which.
  • 4. Event-driven systems: queues, retries, DLQs, idempotency, ordering, fanout.
  • 5. CI/CD + observability + incident response: how you operate production, not just deploy it.
Core Concepts To Learn
1
Cloud provider + Terraform depth
Mandatory
Know
  • One major cloud in production terms: networking, IAM, compute, storage, reliability, and cost.
  • Terraform modules, remote state, per-environment isolation, variables, and plan/apply workflow.
  • How to explain design decisions, not just resource names.
Be ready to answer
  • How do you structure Terraform for staging and production safely?
  • How do you prevent drift and dangerous applies?
2
Containers and serverless workloads
Mandatory
Know
  • Docker fundamentals, image lifecycle, health checks, and resource limits.
  • Cloud Run vs Kubernetes vs ECS style tradeoffs.
  • When serverless is enough and when orchestration control is needed.
Be ready to answer
  • Why would you choose Cloud Run for one service and Kubernetes for another?
  • What scaling or operational constraints push you away from serverless?
3
Event-driven architecture
Mandatory
Know
  • Topics, publishers, subscribers, queues, fanout, retries, backoff, and dead-letter queues.
  • At-least-once delivery, duplicate messages, idempotency, and ordering tradeoffs.
  • Pub/Sub, SQS/SNS, or EventBridge equivalents at concept level.
Be ready to answer
  • How do you design consumers so retries do not create duplicate side effects?
  • When do you add a DLQ and what do you monitor on it?
4
CI/CD and release safety
Mandatory

  • Secret handling, approvals, artifact immutability, and deployment traceability.
  • Reliability and speed tradeoffs in pipelines.
Be ready to answer
  • How do you make infrastructure deploys safe in CI/CD?
  • What should happen before production promotion?

Observability and distributed systems
Mandatory

  • What would you alert on for a critical production API?

Architecture, scalability, and cost ownership
Mandatory

  • How to explain why one architecture is better for a workload than another.
Be ready to answer

  • What data would you gather before changing architecture?
