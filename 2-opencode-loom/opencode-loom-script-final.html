Topic 1 — Cluster Configuration and Scaling
⚙️ Cluster tiers, auto-scaling, multi-region, Terraform MANDATORY

What you need to know: Atlas clusters come in tiers — M0 (free), M10/M20 (dev), M30+ (production). For production you need at minimum M30 for replica set elections. You configure cluster tier, region, cloud provider (GCP/AWS/Azure), and MongoDB version at provision time.

Auto-scaling: Atlas can auto-scale both compute (cluster tier) and storage independently. You set a min/max tier range and Atlas scales within it based on CPU and memory utilization. Key setting: autoScaling.compute.enabled = true.

Multi-region clusters: You define electable nodes (can become primary) and read-only nodes across regions. A 3-region setup with priority gives you geo-redundancy — primary election stays in the highest-priority region.

Terraform: The mongodbatlas_advanced_cluster resource is the current way. Key fields: cluster_type (REPLICASET/SHARDED), replication_specs with region_configs per region, electable_specs and read_only_specs per region.

resource "mongodbatlas_advanced_cluster" "prod" {
  project_id   = var.atlas_project_id
  name         = "prod-cluster"
  cluster_type = "REPLICASET"
  replication_specs {
    region_configs {
      provider_name = "GCP"
      region_name   = "EUROPE_WEST_1"
      priority      = 7
      electable_specs { instance_size = "M30" node_count = 3 }
    }
  }
}
Practice: Sign up for a free Atlas account (M0). Provision a cluster via UI, then write Terraform to match it. Run terraform plan and verify the diff is zero.

Q: How would you scale an Atlas cluster during a traffic spike without downtime?
A: Enable auto-scaling with a compute range (e.g. M30–M60). Atlas detects sustained high CPU/memory and scales up with zero downtime via rolling restarts. For predictable spikes, manually scale via API or Terraform before the event. Storage auto-scales independently and never causes downtime.
Topic 2 — Replication and Failover
🔄 Replica sets, elections, priority, failover testing MANDATORY

Replica sets: Atlas always runs replica sets — minimum 3 nodes (1 primary, 2 secondaries). The primary receives all writes. Secondaries replicate asynchronously via the oplog. If primary fails, secondaries hold an election — the node with highest priority and most up-to-date oplog wins.

Elections: Require a majority of voting members. In a 3-node set, 2 nodes must agree. If you lose 2 nodes you lose write availability. This is why 3-node minimum matters — losing 1 still leaves a quorum of 2.

Hidden and delayed members: Non-electable nodes used for analytics reads (hidden=true, priority=0) or point-in-time recovery (delayed member with N-hour lag). Atlas analytics nodes are this pattern.

Read preferences: primary (default, always reads from primary), primaryPreferred, secondary (read scaling but may see stale data), nearest (lowest latency). Application sets this in connection string: readPreference=secondaryPreferred.

Testing failover: Atlas UI has a "Test Failover" button — it steps down the primary and times the election. Typically under 30 seconds. You can also run db.adminCommand({replSetStepDown: 60}) in mongosh.

Practice: On your free Atlas M0, open mongosh, run rs.status() and read every field. Understand stateStr, optimeDate, lastHeartbeat. Then trigger a test failover via the UI and watch the election in rs.status() output.

Q: What happens to writes during a primary election in MongoDB?
A: Writes are rejected with a NotWritablePrimary error during the election window (typically 10–30 seconds). Applications should implement retry logic — the MongoDB drivers handle this automatically with retryWrites=true in the connection string, which is the default on Atlas.
Topic 3 — Backup and Disaster Recovery
💾 Cloud Backup, PITR, snapshot schedules, restore testing MANDATORY

Cloud Backup (current): Atlas takes snapshots of your cluster's storage volumes. You configure a schedule: hourly, daily, weekly, monthly. Each has a retention period. Snapshots are stored in Atlas-managed cloud storage in your cluster's region.

Point-in-Time Recovery (PITR): Atlas streams the oplog continuously alongside snapshots. This lets you restore to any second within the PITR window (up to 7 days). You pick a snapshot as the base, then replay the oplog to your chosen timestamp.

Backup schedule via Terraform:

resource "mongodbatlas_cloud_backup_schedule" "prod" {
  project_id   = var.project_id
  cluster_name = mongodbatlas_advanced_cluster.prod.name
  reference_hour_of_day    = 3
  reference_minute_of_hour = 0
  restore_window_days      = 7
  policy_item_hourly {
    frequency_interval = 6
    retention_unit     = "days"
    retention_value    = 7
  }
  policy_item_daily {
    frequency_interval = 1
    retention_unit     = "days"
    retention_value    = 30
  }
}

RTO and RPO: Know these terms cold. RPO (Recovery Point Objective) = how much data you can afford to lose. RTO (Recovery Time Objective) = how long recovery takes. With PITR enabled, RPO is seconds. RTO depends on database size — Atlas restores to a new cluster, not in-place.

mongodump / mongorestore: For self-hosted or offline exports. mongodump --uri="mongodb+srv://..." creates BSON dumps. mongorestore loads them back. Not the primary Atlas backup mechanism but good to know as a fallback.

Practice: On Atlas M0, enable Cloud Backup. Take a manual snapshot via UI. Then go through the restore flow (you don't have to complete it — just understand every option presented). Write the Terraform for a backup schedule.

Q: How would you design a backup strategy for a production MongoDB Atlas cluster with an RPO of 1 hour?
A: Enable Cloud Backup with hourly snapshots retained for 7 days, daily snapshots for 30 days. Enable PITR with a 7-day window — this actually gives sub-second RPO. Test restores quarterly to a separate cluster to validate RTO. Document the runbook including who approves a restore and what validation steps confirm data integrity post-restore.
Topic 4 — Performance Tuning and Indexing
⚡ Performance Advisor, explain(), indexes, slow queries IMPORTANT

Atlas Performance Advisor: Automatically surfaces slow queries (over 100ms by default) and suggests missing indexes. It shows the query shape, how many times it ran, average execution time, and the suggested index. This is your first stop for performance issues — check it before anything else.

explain(): Run db.collection.find({field: value}).explain("executionStats"). Key fields to read: stage — COLLSCAN means no index (bad), IXSCAN means index used (good). nReturned vs totalDocsExamined — if examined >> returned, the index is inefficient or missing.

Index types: Single field {field: 1}, compound {a: 1, b: -1} (order matters — follows ESR rule: Equality first, Sort second, Range third), text indexes for full-text search, TTL indexes for auto-expiry of documents, partial indexes to index only a subset of documents.

Creating indexes without blocking production: In older MongoDB you'd use {background: true}. In MongoDB 4.2+ all index builds are non-blocking by default on secondaries, then rolled to primary. On Atlas this is handled automatically.

Connection pool sizing: Key for DevOps. Too many connections overwhelm the DB. Atlas shows connection count in metrics. Configure your application's connection pool max — for a 3-node M30 cluster, total connections limit is ~3000. Size pools per service accordingly.

Practice: On your M0 cluster, insert 10,000 documents with a script, run a query without an index, run explain() and see COLLSCAN. Then create an index and re-run explain() — see IXSCAN. This takes 20 minutes and makes the concept permanent.

Q: A query that was running in 50ms is now taking 3 seconds after a data growth. How do you investigate?
A: First check Atlas Performance Advisor — it will likely show the query shape as a slow query. Run explain("executionStats") on the query. If stage is COLLSCAN, a required index is missing or not being used. Check if the collection scan count grew with data — a COLLSCAN that was fast at 10K docs is slow at 10M. Create the suggested compound index (non-blocking on Atlas), verify with explain() that stage becomes IXSCAN, confirm query time drops.
Topic 5 — Security and Access Management
🔒 Network access, RBAC, auth, encryption, VPC peering MANDATORY

Network access: Atlas blocks all connections by default. You allowlist IP ranges or configure VPC Peering / Private Link. For production, never use 0.0.0.0/0. Use VPC Peering to your GCP VPC so only internal cluster traffic reaches Atlas — no public internet exposure.

VPC Peering via Terraform:

resource "mongodbatlas_network_peering" "gcp" {
  project_id    = var.project_id
  container_id  = mongodbatlas_network_container.primary.id
  provider_name = "GCP"
  gcp_project_id = var.gcp_project
  network_name   = var.gcp_network
}

Database users and RBAC: Atlas has built-in roles: readWrite, dbAdmin, clusterMonitor, atlasAdmin. Create scoped users per service — your API service gets readWrite on its DB only, your monitoring agent gets clusterMonitor. Never use the admin user in application code.

Authentication methods: SCRAM (username/password, default), x.509 client certificates, AWS IAM, LDAP. For production at scale, x.509 or IAM is preferred over passwords — no secrets to rotate.

Encryption: At-rest encryption is on by default on Atlas paid tiers using cloud provider KMS (GCP CMEK, AWS KMS). In-transit is TLS 1.2+ enforced — connection strings use mongodb+srv:// which enables TLS automatically. You can bring your own KMS key via mongodbatlas_encryption_at_rest Terraform resource.

Audit logging: Atlas can log all auth attempts, CRUD operations, and admin actions. Route to your SIEM. Configure via mongodbatlas_auditing resource.

Practice: On Atlas M0: create a new DB user with readWrite only on one specific database. Test that it cannot access other databases. Then try to connect without allowlisting your IP — confirm connection refused. Write the Terraform for both resources.

Cloud Infrastructure + Terraform
☁️ Terraform — modularisation, remote state, multi-environment MANDATORY · YOUR STRENGTH

Modularisation: Break infrastructure into reusable modules — networking, compute, IAM, database. Each module has variables.tf, outputs.tf, main.tf. Caller passes variables. Modules live in a shared registry or local modules/ directory. This is what allows you to stamp out identical environments with different var files.

Remote state: Store terraform.tfstate in GCS or S3, never locally. Add state locking via DynamoDB (AWS) or GCS native locking. Backend config:

terraform {
  backend "gcs" {
    bucket = "my-tfstate-prod"
    prefix = "services/api"
  }
}

Multi-environment setups: Two patterns — workspace-based (terraform workspace new staging) or directory-based (envs/prod/, envs/staging/). Directory-based is safer for production because workspaces share a backend bucket and accidental cross-env applies are possible.

Q: How do you manage Terraform state across multiple teams safely?
A: Remote state in GCS/S3 with state locking enabled. Split state files per service/team — one tfstate per bounded domain, not one monolithic state. Use Terraform Cloud or Atlantis for plan/apply in CI with PR-based review workflow. This prevents concurrent applies and gives full audit trail.
🏗️ GCP — Cloud Run, GKE, networking, IAM, cost optimisation YOUR STRENGTH

Cloud Run: Fully managed serverless container platform. Scales to zero, scales on request concurrency. Key config: --concurrency (requests per container instance), --min-instances (avoid cold starts), --max-instances (cost cap). Deploy via: gcloud run deploy --image gcr.io/project/image --region us-central1. Use with VPC Connector for private network access.

IAM principle of least privilege: Workload Identity for GKE workloads — bind a K8s service account to a GCP service account so pods get GCP permissions without key files. This is the production pattern — no JSON keys in containers.

Cost optimisation: Committed Use Discounts (CUDs) for predictable workloads, Preemptible/Spot instances for batch, right-sizing recommendations from GCP Recommender, Cloud Run min-instances=0 for low-traffic services.

Q: How would you architect a cost-optimised GCP infrastructure for a high-scale SaaS platform?
A: Cloud Run for stateless API services (scales to zero, no idle cost), GKE Autopilot for workloads needing more control (node management abstracted, pay per pod), Cloud SQL or Atlas for DB (managed, no VM overhead), GCS for storage. CUDs on predictable baseline compute. Budget alerts at 80% and 100% of monthly target. Cost allocation labels on every resource so you can attribute spend per team/service.
Event-Driven Architecture
📨 Pub/Sub, SQS/SNS, EventBridge — patterns and ownership GAP TO ADDRESS

Core pattern: Producer publishes an event to a topic. One or more consumers subscribe and process independently. Producers and consumers are decoupled — neither knows the other exists. This is how you scale services without tight coupling.

GCP Pub/Sub specifics: Topics hold messages. Subscriptions pull or push messages to consumers. Messages have a 7-day retention max. At-least-once delivery — consumers must be idempotent (processing the same message twice should have no side effects). Acknowledgement deadline — if consumer doesn't ack within the deadline, message is redelivered. Dead-letter topic — messages that fail N times go here for inspection.

DevOps ownership of event-driven systems: Provision topics and subscriptions via Terraform (google_pubsub_topic, google_pubsub_subscription). Monitor: undelivered message count (if rising, consumers are falling behind), oldest unacked message age (if high, consumer is stuck). Set alerts on both. DLQ (dead letter queue) alert is critical — any message in DLQ means a consumer failed permanently.

Your ServeLLM angle: "I designed the internal service communication with async patterns — decoupled services, retry logic, failure handling — which is the same operational model as Pub/Sub. The infrastructure ownership pattern is identical: provision the messaging layer as code, monitor queue depth and DLQ, set alerts for consumer lag."

Q: How do you ensure message ordering and exactly-once processing in Pub/Sub?
A: GCP Pub/Sub supports message ordering via ordering keys — messages with the same key are delivered in order to a subscription. For exactly-once processing, you use Pub/Sub's exactly-once delivery feature (available on pull subscriptions) combined with idempotent consumer logic. In practice, design consumers to be idempotent first — then ordering and delivery guarantees become less critical to correctness.

CI/CD Pipelines
🚀 GitHub Actions, Cloud Build, GitOps with ArgoCD YOUR STRENGTH

GitHub Actions pattern for infra: PR triggers terraform plan — output posted as PR comment. Merge to main triggers terraform apply. Separate workflows per environment. Secrets via GitHub Environments with environment-scoped protection rules.

Cloud Build (GCP-native): Defined in cloudbuild.yaml. Triggered by push to Cloud Source Repositories or GitHub via Cloud Build triggers. Steps run in containers — each step is a Docker image. Artifacts stored in GCS or Artifact Registry. Key advantage over GitHub Actions for GCP: runs inside your GCP project, no external secret exposure, native IAM auth.

GitOps with ArgoCD: Git is the source of truth for Kubernetes state. ArgoCD watches a Git repo and syncs cluster state to match. Deviation detection — if someone does kubectl apply manually, ArgoCD detects drift and alerts or auto-reverts. Application defined as an Application CR pointing at a Helm chart or Kustomize directory.

Q: How do you prevent a bad Terraform apply from taking down production?
A: Plan in CI on PR — never apply without a reviewed plan. Separate state per environment — staging and prod are isolated, staging apply can't affect prod. Use Sentinel or OPA policies to block dangerous patterns (e.g. destroying a database, removing security groups). Maintain a rollback runbook — Terraform doesn't auto-rollback, so know your manual recovery path. Enable drift detection so you catch out-of-band changes before the next apply.
Monitoring, Logging, Observability
📊 Prometheus/Grafana, Datadog, APM, distributed tracing PARTIAL GAP — Datadog

Your strength: Prometheus + Grafana + OpenTelemetry — you've done this in production. Be ready to explain SLI/SLO definition: SLI is the metric (e.g. p99 latency), SLO is the target (e.g. p99 < 200ms 99.9% of the time), error budget is the allowed violation window.

Datadog (gap to address): Agent-based — Datadog Agent runs as a DaemonSet on K8s nodes, scrapes container metrics, logs, and traces. Key features: APM (auto-instrumented traces from your app), Log Management (structured log ingestion with tag-based search), Infrastructure Maps (visual service dependency), Monitors and Alerts (more powerful than raw Alertmanager). For interviews, know that Datadog replaces Prometheus+Grafana+Jaeger with a single managed platform — the trade-off is cost vs operational overhead.

Distributed tracing: OpenTelemetry is the standard. Each service generates a trace with spans. Trace context propagates via HTTP headers (traceparent). All spans with the same trace ID form a waterfall showing exactly where latency occurs across services. For DevOps, you provision the collector and backend (Jaeger, Tempo, or Datadog APM) — the app team instruments the code.

The SRE framing the JD wants: Define SLOs for each service. Calculate error budget. When budget is nearly exhausted, freeze feature work and focus on reliability. This is a cultural/process concept as much as a technical one.

Q: How do you set up observability for a new microservice from scratch?
A: Four signals — metrics, logs, traces, alerts. Metrics: expose a /metrics endpoint, scrape with Prometheus (or Datadog Agent), build a Grafana/Datadog dashboard for the four golden signals (latency, traffic, errors, saturation). Logs: structured JSON logs with correlation IDs, shipped to log aggregation. Traces: OpenTelemetry SDK in the app, traces to Jaeger or Datadog APM. Incident Response and Architecture Ownership
🚨 Incident response, RCA, runbooks, architecture decisions YOUR STRENGTH — frame it right

This JD is explicit: "Own infrastructure decisions end-to-end" and "lead incident response and root cause analysis." This is not a ticket-executor role. You need to demonstrate architect-level thinking.

Incident response process: Detect (alert fires) → Acknowledge (someone owns it) → Mitigate (stop the bleeding — rollback, redirect traffic, increase replicas) → Investigate (root cause) → Fix (permanent solution) → Post-mortem (blameless, document, prevent recurrence).

Runbooks: Pre-written procedures for known failure modes. "If PgBouncer connection pool is exhausted: 1. Check pool stats via SHOW POOLS. 2. Identify top connection consumers. 3. Temporarily increase pool size. 4. Investigate and fix the connection leak." Write these before incidents, not during.

Your ServeLLM angle for architecture ownership: "At Techanzy I owned the full infrastructure stack — I made the decision to use PgBouncer in transaction mode rather than session mode, I chose the connection pool sizing, I designed the health check strategy. When we had a connection pool exhaustion incident I was the one who diagnosed and resolved it — and then wrote the runbook so it never became a firefight again."

Q: Tell me about a production incident you owned and resolved.
A: Use the ServeLLM PgBouncer connection pool exhaustion story if asked. Structure: what failed (connections rejected, backend unhealthy), how you detected it (health check alert + Grafana spike in connection wait time), how you mitigated (temporarily bumped pool size via config reload, no restart needed), root cause (a background job was not releasing connections on exception — connection leak), permanent fix (added finally block + connection validation in the job), prevention (added pool saturation alert at 80% threshold so next time it fires with 20% headroom).
Containerisation and Kubernetes
🐳 Docker, GKE, Kubernetes workloads, Helm, security YOUR STRENGTH

Kubernetes production essentials: Resource requests and limits on every pod (without limits one noisy neighbour can starve others). PodDisruptionBudgets to ensure rolling updates don't take down too many replicas. HorizontalPodAutoscaler on CPU/memory or custom metrics. Network Policies to restrict pod-to-pod traffic (zero-trust within the cluster).

GKE Autopilot vs Standard: Autopilot abstracts node management — you pay per pod resource request, GKE manages nodes. Standard gives you full node control — you manage node pools, OS, autoscaling. For this role, knowing the trade-offs matters more than which is "better."

Kubernetes RBAC: ServiceAccounts for workload identity, Roles/ClusterRoles for permission sets, RoleBindings to attach them. Principle: every pod runs as a non-root ServiceAccount with only the permissions it needs. Audit RBAC drift regularly — kubectl auth can-i --list --as=system:serviceaccount:namespace:sa-name.

Q: How do you harden a GKE cluster for a production SaaS workload?
A: Workload Identity instead of service account keys. Network Policies (deny-all default, allow only required paths). PodSecurityStandards enforcement (restricted profile — no root, no privileged containers). Private cluster (nodes have no public IPs, control plane access via authorized networks). Binary Authorization — only signed images from Artifact Registry can deploy. RBAC with least-privilege ServiceAccounts. Node auto-upgrade enabled for security patches. This is all Terraform — no manual kubectl.
