HR154 · Updated JD Delta

Mongo Open

The updated JD moved MongoDB Atlas from a supporting requirement to the main gap you need to close. Learn the five Mongo topics below in this order and be ready to explain them at owner level, not tool-user level.

Original JD

MongoDB Atlas was present, but as one item inside a broader GCP, Cloud Run, Pub/Sub, Terraform role.

Updated JD

Mandatory deep MongoDB Atlas experience is now central: cluster scaling, replication/failover, backup/recovery, performance/indexing, and security/access.

Main takeaway: Pub/Sub and Cloud Run are no longer the differentiator. MongoDB Atlas ownership is.
Your angle: be honest that your production DB depth is stronger in PostgreSQL, then show Atlas IaC knowledge, distributed DB fundamentals, and hands-on practice proof.
Exact Learning Order
1
Atlas cluster setup and scaling
Most important start
Learn
  • Cluster tiers: M0 for practice, M30+ for real production conversations.
  • REPLICASET vs sharded clusters.
  • Region, cloud provider, MongoDB version, auto-scaling.
  • mongodbatlas_advanced_cluster in Terraform.
Do
  • Create one Atlas account and one free cluster.
  • Write Terraform to match the cluster config.
  • Be able to explain how you would scale from M30 to higher tiers without downtime.
2
Replication, elections, and failover
Mandatory
Learn
  • Primary, secondary, oplog, quorum, elections.
  • What happens to writes during primary failover.
  • Read preferences and stale reads.
Do
  • Connect with mongosh.
  • Run rs.status() and understand stateStr, optimeDate, and heartbeat fields.
  • Trigger Atlas test failover and note election timing.
3
Backups, PITR, and disaster recovery
Mandatory
Learn
  • Snapshots vs Point-in-Time Recovery.
  • RPO and RTO.
  • Restore flow and the fact Atlas restores to a new cluster.
Do
  • Take a manual snapshot.
  • Go through restore options in Atlas UI.
  • Write a Terraform backup schedule resource and explain hourly plus daily retention.
4
Performance tuning and indexing
Mandatory
Learn
  • Atlas Performance Advisor.
  • explain("executionStats").
  • COLLSCAN vs IXSCAN.
  • Single-field, compound, TTL, and partial indexes.
Do
  • Insert test data.
  • Run one query without an index, then with an index.
  • Be able to explain why data growth makes unindexed queries suddenly slow.
5
Security, network, and access control
Mandatory
Learn
  • IP allowlist vs VPC peering / PrivateLink.
  • Least-privilege database users and Atlas roles.
  • TLS, encryption at rest, KMS, audit logs.
Do
  • Create one scoped DB user for one database only.
  • Test failed access outside that scope.
  • Be ready to say why 0.0.0.0/0 is bad for production and why peering is better.
6
Package your interview story
Do last
Say this clearly
"My strongest production database background is PostgreSQL, but for MongoDB Atlas I focused on the infrastructure and operations layer: cluster setup, Terraform automation, failover, backup strategy, indexing basics, and secure connectivity. The core reliability concepts transfer directly, and I have been practicing Atlas hands-on against those exact areas from the JD."
Be able to answer
  • How do you scale an Atlas cluster?
  • What happens during primary election?
  • How would you design backup plus restore testing?
  • How do you investigate a slow query?
  • How do you secure Atlas for production SaaS?
7-Day Sprint
  • Day 1: Create Atlas cluster, connect with mongosh, browse Atlas UI.
  • Day 2: Write Terraform for cluster, DB user, and IP access list.
  • Day 3: Learn replica set flow, run rs.status(), trigger test failover.
  • Day 4: Configure backup concepts, snapshot flow, PITR, restore path.
  • Day 5: Insert sample data, test explain(), create indexes, compare results.
  • Day 6: Create scoped users, review network security, encryption, audit logging.
  • Day 7: Rehearse the five mandatory answers out loud until they sound natural.
Do Not Overfocus
  • Do not spend most of your time on app-side Mongo schema design.
  • Do not spend most of your time on Pub/Sub now.
  • Do not claim deep production Atlas ownership unless you can defend it with hands-on practice.
Updated JD · Full Learning Scope

All Concepts

This tab is the full map of what to learn from updated-jd-with-mongo.md. MongoDB Atlas is still the highest-priority gap, but the JD also expects strong Terraform, cloud ownership, containers, event-driven systems, CI/CD, observability, incident response, and async engineering maturity.

Top Priority

MongoDB Atlas plus the ability to speak at architect-owner level.

Second Priority

Terraform + cloud systems design, then containers, event-driven systems, CI/CD, and observability.

Important: the updated JD is not asking for only tool knowledge. It wants ownership, architecture judgment, performance thinking, and operational maturity.
Use this tab like a checklist: if you cannot explain a card below clearly out loud, that concept is still a gap.
Priority Order
  • 1. MongoDB Atlas: cluster scaling, replication, failover, backups, indexing, security.
  • 2. Terraform + cloud ownership: modularisation, remote state, environment separation, design tradeoffs.
  • 3. Containers + serverless: Docker, Kubernetes, Cloud Run or equivalents, when to choose which.
  • 4. Event-driven systems: queues, retries, DLQs, idempotency, ordering, fanout.
  • 5. CI/CD + observability + incident response: how you operate production, not just deploy it.
Core Concepts To Learn
1
Cloud provider + Terraform depth
Mandatory
Know
  • One major cloud in production terms: networking, IAM, compute, storage, reliability, and cost.
  • Terraform modules, remote state, per-environment isolation, variables, and plan/apply workflow.
  • How to explain design decisions, not just resource names.
Be ready to answer
  • How do you structure Terraform for staging and production safely?
  • How do you prevent drift and dangerous applies?
2
Containers and serverless workloads
Mandatory
Know
  • Docker fundamentals, image lifecycle, health checks, and resource limits.
  • Cloud Run vs Kubernetes vs ECS style tradeoffs.
  • When serverless is enough and when orchestration control is needed.
Be ready to answer
  • Why would you choose Cloud Run for one service and Kubernetes for another?
  • What scaling or operational constraints push you away from serverless?
3
Event-driven architecture
Mandatory
Know
  • Topics, publishers, subscribers, queues, fanout, retries, backoff, and dead-letter queues.
  • At-least-once delivery, duplicate messages, idempotency, and ordering tradeoffs.
  • Pub/Sub, SQS/SNS, or EventBridge equivalents at concept level.
Be ready to answer
  • How do you design consumers so retries do not create duplicate side effects?
  • When do you add a DLQ and what do you monitor on it?
4
CI/CD and release safety
Mandatory
Know
  • Build, test, plan, deploy, rollback, and environment promotion flow.
  • Secret handling, approvals, artifact immutability, and deployment traceability.
  • Reliability and speed tradeoffs in pipelines.
Be ready to answer
  • How do you make infrastructure deploys safe in CI/CD?
  • What should happen before production promotion?
5
Observability and distributed systems
Mandatory
Know
  • Metrics, logs, traces, dashboards, alerting thresholds, and latency analysis.
  • Golden signals: latency, traffic, errors, saturation.
  • Advanced tools named by the JD: Datadog, APM, tracing.
Be ready to answer
  • How do you investigate an intermittent latency spike across services?
  • What would you alert on for a critical production API?
6
Architecture, scalability, and cost ownership
Mandatory
Know
  • Capacity planning, bottleneck identification, scaling patterns, and cost-performance tradeoffs.
  • What "owner level" means: you choose, justify, monitor, and improve systems.
  • How to explain why one architecture is better for a workload than another.
Be ready to answer
  • How do you balance performance and cost for a high-scale SaaS platform?
  • What data would you gather before changing architecture?
7
Incident response and root cause analysis
Mandatory
Know
  • Triage, mitigation, communication, timelines, impact assessment, and follow-up actions.
  • The difference between symptoms, trigger, root cause, and contributing factors.
  • How runbooks reduce recovery time.
Be ready to answer
  • How do you handle a live production outage while keeping stakeholders updated?
  • What should a good RCA contain after the incident is resolved?
8
Documentation, async communication, and mentoring
Mandatory
Know
  • Runbooks, architecture docs, decision records, and operational best practices.
  • What strong remote async communication looks like: clear status, written decisions, crisp handoffs.
  • How senior engineers mentor through reviews, standards, and system thinking.
Be ready to answer
  • How do you document a system so another engineer can operate it safely?
  • How do you raise team quality without becoming a bottleneck?
Nice To Have Concepts
  • Multi-cloud: be able to compare GCP, AWS, and Azure patterns at a high level.
  • GitOps: understand pull-based deploys, desired state, and drift visibility.
  • Advanced observability: Datadog, APM, distributed tracing.
  • High-scale SaaS: noisy neighbors, tenant isolation, shared vs dedicated infrastructure.
  • Platform engineering: internal tooling that makes developer workflows safer and faster.
How To Use This Tab
  • If a card is already your strength, rehearse how to explain it cleanly.
  • If a card is weak, learn the core vocabulary, one real example, and one failure mode.
  • For MongoDB, go back to the first tab because that is the main delta in the updated JD.
