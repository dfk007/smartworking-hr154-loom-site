Design system first

Make fewer, stronger decisions.

Premium UI comes from consistency, not novelty. Define the system before the layout so every component feels like it belongs to the same page.

Typography pairing

  • Use one serif for large hero and section headings.
  • Use one sans font for UI and body text.
  • Use typography contrast to establish hierarchy immediately.

Color rules

  • Keep the base dark and neutral.
  • Use only 2 to 3 accent colors.
  • Use accents as highlights, not dominant fills.

Surface language

  • Translucent dark surfaces.
  • Thin bright borders.
  • Large radii and deep soft shadows.

Production order

Use the workflow, not random iteration.

The fastest way to get better pages is to follow the same order every time instead of jumping between styling, animation, and content without a system.

  1. Identify the page type.
  2. Identify the conversion goal.
  3. Generate a design system first.
  4. Decide deployment constraints early.
  5. Create the content hierarchy.
  6. Build the hero first.
  7. Create reusable visual primitives.
  8. Add motion after the layout works.
  9. Make dense content easier to read.
  10. Build responsive behavior intentionally.
  11. Add reduced-motion support.
  12. Validate for deployment reality.
  13. Format and publish.

Prompt kit

Use for long guide pages

Ask for a premium static guide page with chapter tabs, dense content panels, polished tables, callouts, and media embeds.

Always add this constraint

If the final page will be deployed as a plain HTML file on GitHub Pages, do not rely on React or Framer Motion for the visible content.

Static blueprint

Keep the architecture simple and repeatable.

The blueprint focuses on a shell container, reusable panels, hero-first section order, and small local JavaScript for narrow interactions like tabs and accordions.

  • Use a centered max-width shell container.
  • Use one panel recipe across cards and surfaces.
  • Use a hero, overview layer, content sections, and final CTA.
  • Use local JS only for small interactions.
  • Use CSS for motion and responsiveness.

Deployment reality

Pages are only finished when they survive deployment.

The most common failure mode is a page that looks good locally but breaks after push because rendering depended on runtime imports or incorrect paths.

Deployment rules

  • Prefer static HTML for plain GitHub Pages.
  • Prefer relative links.
  • Prefer local JavaScript only.
  • Avoid runtime dependencies for core rendering.

QA checks

  • Hero looks strong on desktop and mobile.
  • Panels are visually consistent.
  • Hover and focus states exist.
  • Reduced motion is respected.

Common fixes

  • Blank page: move visible content into static HTML.
  • Broken deploy: verify paths and hosting assumptions.
  • Generic look: define the design system first.

When to stay static-first

  • The project is deployed as plain HTML files.
  • GitHub Pages is the main host.
  • The page only needs local tabs, accordions, or toggles.
  • You want deterministic rendering with minimal failure points.

When to use Framer Motion instead

  • The project is a real React/Vite/Next app.
  • A build process already exists.
  • The deployment target supports the built app.
  • Motion meaningfully improves the experience.
